## 📄 문서 2. 시스템 설계 문서

- [ ] **문제 정의 (업무 과업 선택)**
  - 과업 1개 선택 (메일 초안 / 회의록 요약 / 보고서 개요 / 요구사항 정리 등)
- [ ] **타겟 사용자** — 누가 쓰는가
- [ ] **입력 데이터 형태** — 어떤 정보를 받는가
- [ ] **출력 규격** — 어떤 형식으로 내는가
- [ ] **입력 템플릿 (재사용 가능)** — 예: `[목적] [대상] [핵심포인트] [톤] [금지어] [필수항목]`
- [ ] **페르소나 정의**
  - 이름 / 역할(직무) / 전문 분야 / 말투 / 금지 사항 / 우선순위(정확성 vs 친절함)
- [ ] **시스템 프롬프트 (필수 요소 포함)**
  - 역할 + 목표
  - 출력 형식 규칙
  - 안전장치 (모르면 모른다고, 확인 질문)
  - 사실/수치/정책 처리 규칙 (근거 제시 or 확인 필요 표기)
- [ ] **Few-shot 예시 3개 이상**
  - 그중 **1개는 반드시 "모호한 입력"** → AI가 되묻는 케이스
- [ ] **v1 → v2 개선 이력**
  - v1: 간단 지시 프롬프트
  - v2: 단계적 추론 유도 (요구사항 정리 → 누락 질문 → 초안)
  - ※ 최종 답변엔 추론 과정 노출 X, "핵심 근거 bullet 중심" 규칙 포함
- [ ] **환각 검증 설계**
  - 사실 기반 검증 질문 **5개 이상**
  - 각 질문마다: 기대 정답(판정 기준) / 허용 오차 / 참고 근거
  - Pass·Fail 기준 명시
- [ ] **최종 프롬프트 전문**

## 📄 문서 3. 실행 로그 (대화 로그)

- [ ] **시나리오 설명**
- [ ] **10턴 이상 대화 전문**
  - 아래 중 1개 이상 반드시 포함:
    - 중간 조건 변경 (톤/대상/정책)
    - 뒤늦은 추가 정보 제공
- [ ] **문맥 유지 검증** — 확정 요구사항이 10턴 내내 유지되었는지, 변경 시 해당 부분만 반영했는지
- [ ] **문제 발생 지점** — 환각 / 문맥 끊김 등
- [ ] **수정 결과 요약**

## ✅ 작성 순서 추천

1. **과업 선택 + 문제 정의** (문서 2 앞부분) → 모든 것의 출발점
2. **페르소나 + 시스템 프롬프트 + Few-shot** 설계
3. **동일 입력으로 3개 모델 테스트** → 문서 1 작성
4. **환각 검증 5문항 실행** → v1/v2 개선 반영
5. **10턴 대화 수행** → 문서 3 로그 정리

---

작업 규모가 크지만, **"과업 하나를 잘 정의하면 나머지가 연쇄적으로 채워지는" 구조**예요. 과업부터 정하면 훨씬 수월합니다. 💪
